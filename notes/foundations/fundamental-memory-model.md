## 一、Core C++ Fundamentals（核心 C++ 基础）

### 1. 编译与内存模型

**C++ compilation model**

源文件变成可执行文件要经过三步：预处理（展开 `#include`、宏、条件编译）→ 编译（每个 .cpp 单独编译成 .o/.obj，此时只知当前翻译单元）→ 链接（把多个目标文件与库合并成一个可执行文件）。理解这一点很重要：**单一定义规则 (ODR)** 要求非 inline 的全局变量、函数、类成员等在整份程序中只能有一处定义，所以通常把声明放在头文件、定义放在 .cpp，避免重复定义。模板比较特殊：模板在实例化时才生成代码，所以模板的定义往往放在头文件里，或者用显式实例化。编译模型还影响增量编译、链接错误（undefined reference、multiple definition）的排查，以及静态/动态库的用法。

**Stack vs Heap**

栈是线程私有的、由编译器与运行时自动管理的一块连续内存。函数调用时局部变量、参数、返回地址等压栈，函数返回时自动弹栈，无需手写释放。栈大小通常有限（如几 MB），分配和释放只是移动栈指针，速度极快，且地址连续、缓存友好。堆是进程内共用的动态内存区，通过 `new`/`malloc` 向操作系统或分配器申请，通过 `delete`/`free` 或智能指针释放。堆的大小受系统与配置限制，分配可能涉及系统调用、空闲链表或更复杂的分配器，有延迟和碎片问题。在 HFT 或低延迟场景中，热路径上应尽量减少堆分配，优先用栈上对象或预分配池，这样延迟更可预测、也更利于 CPU 缓存。

**Memory layout of a C++ program**

典型进程的虚拟地址空间（自低地址到高地址）大致为：**代码段 (text)** 存机器码；**只读数据 (rodata)** 存 const 常量、字符串字面量；**已初始化数据 (data)** 和 **未初始化数据 (bss)** 存全局变量与静态变量；**堆 (heap)** 向上增长，由 new/malloc 使用；**栈 (stack)** 向下增长，存局部变量与调用栈；再往上可能有共享库、内核空间等。理解布局有助于理解：变量在哪个段、生命周期多长；为什么栈溢出或越界写会破坏相邻数据；为什么连续访问（如数组遍历）比随机访问更友好（缓存行、预取）；以及对齐（alignment）如何影响访问效率和原子操作。

### 2. 类型与语义

**const correctness**

`const` 表示“不修改”：加在变量上表示值不变，加在成员函数后（如 `void foo() const`）表示该函数不修改对象状态，加在引用或指针上表示不能通过该引用/指针修改目标。坚持 const 正确性能减少误改、让接口意图更清晰（只读 vs 可写），并给编译器更多优化空间（如把 const 对象放在只读页）。写代码时建议默认加 const，只有确实需要修改的地方才去掉；参数能传 `const T&` 就尽量传，避免意外修改调用方的对象。

**References vs pointers**

引用是对象的别名，绑定后不能改绑，也不能为 null，语法上像普通变量（不需要解引用）。指针是存储地址的变量，可以重新赋值、可以为 nullptr、可以做算术（如 `p+1`）。引用在 C++ 里常用来做函数参数和返回值：传 `const T&` 避免拷贝又保证不修改；返回引用可以返回“容器内的元素”等，但要避免返回局部变量的引用（悬空引用）。指针适合“可选”或“可重定向”的语义，以及需要算术或与 C 接口交互的场景。两者在底层都可能用地址实现，但类型系统与语义不同，不能混为一谈。

**lvalue / rvalue**

历史上左值指能放在赋值号左边的表达式，右值指只能放在右边的。C++11 后更精确：**左值 (lvalue)** 是有身份（可以取地址）的表达式，如变量、左值引用、返回左值引用的函数调用。**右值 (rvalue)** 一般是临时量或即将被销毁的对象，如字面量、`a+b` 的结果、返回非引用类型的函数调用。判断方法之一：能取地址的是左值，不能取地址的是右值。这个区分是移动语义的基础：右值可以“被搬空”，左值默认不移动（除非用 `std::move` 显式转成右值引用）。重载时 `void f(T&)` 接受左值，`void f(T&&)` 接受右值（或转发），用于实现移动构造、完美转发等。

**move semantics**

移动语义用**右值引用**（`T&&`）和**移动构造/移动赋值**把资源从源对象“偷”过来，避免不必要的深拷贝。例如 `std::vector` 的移动构造只拷贝内部指针和大小，把源对象的指针置空，这样大容器传递或返回时成本是 O(1)。`std::move(x)` 把 `x` 转成右值引用，表示“之后不再使用 x 的当前状态”；移动后源对象处于**有效但未指定状态**，一般只应析构或重新赋值。移动要在类里写 `T(T&&)` 和 `T& operator=(T&&)`，对持有堆内存、文件句柄等的类型非常重要。在热路径上，用移动可以显著减少拷贝和堆分配。

### 3. 资源管理与对象生命周期

**RAII principle**

RAII（Resource Acquisition Is Initialization）是 C++ 管理资源的惯用法：在**构造函数**里获取资源（内存、文件句柄、锁等），在**析构函数**里释放。因为栈上对象在离开作用域时一定会析构，所以只要把资源“绑”在栈对象上，就能保证无论正常返回还是异常，资源都会被释放。标准库里的 `std::fstream`、`std::lock_guard`、智能指针都是 RAII：你不需要手写 `fclose` 或 `unlock`。写自己的类时，若有动态资源，应把释放逻辑放在析构里，并注意拷贝/移动语义（Rule of 3/5），避免重复释放或泄漏。

**Constructors / destructors**

构造函数负责把对象从“未初始化”变成“可用状态”，包括默认构造、拷贝构造、移动构造等；可以重载，没有返回值。析构函数负责清理资源，在对象销毁时自动调用，不能带参数、不能重载。**构造顺序**：先虚基类（按继承顺序），再非虚基类，再成员（按声明顺序），最后派生类自身。**析构顺序**正好相反：先派生类，再成员（逆序），再非虚基类（逆序），再虚基类（逆序）。掌握这个顺序有助于理解多继承、虚继承下的初始化与清理，以及避免“用尚未初始化的成员”或“重复析构”的错误。

**Rule of 3 / 5 / 0**

**Rule of 3**：若类需要自定义**析构函数**（因为要释放资源），通常也需要自定义**拷贝构造函数**和**拷贝赋值运算符**，否则编译器生成的默认拷贝是浅拷贝，会导致双释或共享不该共享的资源。**Rule of 5**：在 C++11 之后，若写了拷贝，往往还要写**移动构造**和**移动赋值**，否则移动会退化成拷贝，失去性能优势。**Rule of 0**：理想情况是类本身不拥有“原始”资源，而是把资源交给智能指针、容器等 RAII 对象管理，这样不需要手写五件套，编译器生成的默认版本就够用。实际项目中尽量往 Rule of 0 靠拢，只在真正管理原始资源时手写 3 或 5。

**Deep vs shallow copy**

**浅拷贝**只复制指针（或句柄），多个对象指向同一块数据；若每个对象析构时都去释放，会重复释放导致未定义行为；若只释放一次，其他对象会变成悬空指针。**深拷贝**会复制指针所指的完整内容，每个对象拥有独立副本，互不影响，但成本高。当类持有动态内存、文件描述符等时，拷贝构造和拷贝赋值应实现深拷贝，或者把该类设为不可拷贝（`= delete`），只提供移动。默认的编译器生成拷贝是浅拷贝，所以有“裸指针成员 + 在析构里 delete”的类必须自己实现或禁用拷贝。

### 4. 类设计：struct / class / 封装

**Struct vs class differences**

在 C++ 里，`struct` 和 `class` 的**唯一**语法区别是默认访问权限：`struct` 默认成员是 `public`，默认继承是 `public`；`class` 默认成员是 `private`，默认继承是 `private`。除此之外（成员函数、继承、模板等）两者完全一致。习惯上，`struct` 常用于“数据为主”的聚合类型（如 POD、配置项、仅含数据的 DTO），`class` 常用于带不变量和复杂行为的类型。按项目风格统一即可；关键是把“谁可以访问什么”想清楚，而不是纠结用哪个关键字。

**Encapsulation**

封装指把**实现细节**藏在 `private` 或 `protected` 里，只通过 `public` 接口与外界交互。这样内部可以随意改（换数据结构、加缓存、改算法），只要接口不变，调用方代码不用改。同时，不暴露内部状态能减少误用（如不让用户直接改破坏不变量）。设计时先想清楚“最小且稳定的 public API”，再把其余都收进 private；对继承，若希望派生类扩展而不希望外部依赖，可用 protected。封装与“高内聚、低耦合”直接相关，是维护和测试的基础。

### 5. 继承与多态

**Inheritance (public/private/protected)**

**public 继承**表示“派生类是一个基类”(is-a)：派生类对象可以当基类对象用，基类的 public 在派生类里仍对调用方可见。这是最常用的多态继承。**protected 继承**下，基类的 public 在派生类里变成 protected，对外不可见；很少用，一般表示“实现继承”且想限制进一步暴露”。**private 继承**表示“用基类实现派生类”(implemented-in-terms-of)：基类的所有成员在派生类里都是 private，外部无法通过派生类访问基类接口。private 继承在语义上常可被“组合”（成员对象）替代，除非需要重写虚函数或占用空基类优化等。

**Virtual functions**

在基类里把函数声明为 `virtual`，派生类中可以**重写 (override)** 该函数。通过基类指针或引用调用时，会根据**实际对象类型**（而非指针类型）决定调用哪个版本，这就是**运行时多态**。实现上，每个多态类型有一个虚表，对象里有一个指向虚表的指针，调用时通过该指针间接跳转。虚函数适合“同一接口、多种实现”的场景（如不同策略、不同设备驱动）。注意：虚函数有调用和无法内联等成本，在极端性能敏感路径要权衡；构造函数和析构函数里调用虚函数时，会按当前构造/析构阶段解析，不会走到派生类。

**vtable internals**

每个包含虚函数的类（或从这样的类继承的类）都有一个**虚表 (vtable)**，表中按顺序存放该类所有虚函数的指针。每个对象在布局最前面（通常）有一个 **vptr**，指向其动态类型的 vtable。调用 `obj.foo()` 时，实际是 `(*(obj.vptr)[foo 的索引])()`。派生类重写时，在派生类的 vtable 里把对应槽位换成派生类的函数地址。多重继承时可能有多份 vptr。理解 vtable 能解释：为什么多态有间接调用成本、为什么不能随意把对象按字节拷贝（vptr 必须正确）、以及 ABI 里对 vtable 布局的约定。

**Virtual destructor importance**

若用**基类指针** `delete` 一个实际指向派生类对象的指针，而基类的析构函数不是虚的，则只会调用基类的析构函数，派生类自己的成员和子对象不会被析构，导致资源泄漏或未定义行为。因此，**只要类设计成会被多态使用（基类指针指向派生类），基类析构函数必须声明为 virtual**。若基类本意不是多态使用，可以把析构设为 `protected` 非虚，这样外部无法 `delete` 基类指针，避免误用。纯虚析构函数可以存在且必须有一份实现（派生类析构会调用它），常用于“抽象基类既要抽象又要能正确析构”的场景。

**Override / final**

在派生类中重写虚函数时，写上 **override** 关键字，编译器会检查是否真的在重写基类的某个虚函数（函数名、参数、const 等一致），否则报错。这样能避免“以为在重写实则签名写错导致变成新虚函数”的 bug。**final** 可以加在虚函数后，表示派生类不能再重写该函数；或加在类名后，表示该类不能再被继承。final 有助于固定设计、避免误继承，有时也能帮助编译器做去虚化优化。

**Object slicing**

当把派生类对象**按值**传给接受基类参数的函数，或赋给基类类型的变量时，只会拷贝/保留基类子对象部分，派生类独有的成员被“切掉”，多态信息丢失，后续再当基类用就只是基类对象。因此，**多态类型应通过指针或引用传递**，不要按值传。若函数需要“保存一份副本”，应考虑克隆接口（如虚 `clone()`）或存智能指针，而不是直接存基类值类型。

**Multiple inheritance**

C++ 允许一个类继承多个直接基类，派生类会包含每个基类的子对象，并可能有多份 vptr。要处理的问题包括：**名字冲突**（两个基类有同名成员时需用 `Base::member` 或类名限定）、**构造/析构顺序**（按继承列表顺序先构造基类再构造派生类）、以及**菱形继承**带来的重复基类子对象和歧义。多重继承要谨慎使用；很多场景用单继承 + 组合或接口继承更清晰。

**Diamond problem**

菱形继承：A 是基类，B 和 C 都继承 A，D 又继承 B 和 C。若 B、C 都是普通继承 A，则 D 里会有**两份** A 的子对象；当 D 访问 A 的成员时，编译器不知道走哪一份，产生歧义。解决办法是让 B 和 C **虚继承** A，这样 D 中只保留一份 A 的子对象，由 D 直接初始化。虚继承会引入额外间接层（通过虚基类指针访问），构造顺序也变复杂（虚基类由最派生类先构造），所以只在确实需要解决菱形时使用。

**Virtual inheritance**

用 `class B : virtual public A` 这样的**虚继承**，表示 A 是“虚基类”；在继承层次中，最终每个 A 只保留一份子对象，由最派生类负责构造。虚继承的实现通常通过对象内的虚基类指针或偏移表访问虚基类成员，多一次间接访问，且构造顺序规则特殊（虚基类先于任何非虚基类）。一般只在菱形结构中必须共享同一基类子对象时才用虚继承，并注意可读性和性能。

**Polymorphism cost analysis**

多态的成本大致包括：每次虚函数调用通过 vptr 查表再间接跳转，比直接调用多一次或多次内存访问；虚函数通常难以内联（除非编译器能 devirtualize）；可能不利于 CPU 分支预测和指令缓存。在极低延迟场景（如 HFT 核心路径），若虚调用在热循环里，可以考虑用 **CRTP**（编译期多态）、**std::variant + std::visit**、或把类型信息编码成枚举/标签避免虚调用，用空间或模板实例化换时间。

**Pure virtual function / interface / abstract class**

**纯虚函数**在声明后加 `= 0`，如 `virtual void foo() = 0;`。含有至少一个纯虚函数的类是**抽象类**，不能实例化，只能作为接口或基类。派生类必须实现所有纯虚函数（否则仍是抽象类）才能实例化。在 C++ 里没有单独的 “interface” 关键字，**接口**通常就是一个只含纯虚函数（和可能还有虚析构）的抽象类。这种用法常见于依赖注入、策略模式、插件系统等“面向接口编程”的场景。

### 6. 其他语言机制

**Inline, static, namespace, etc.**

**inline** 在 C++17 前主要表示“建议编译器内联展开”，编译器可以忽略；在 C++17 后更重要的语义是允许**多个翻译单元**中定义同一个 inline 函数/变量，链接时只保留一份（用于头文件里的函数定义）。**static** 有多义：在文件作用域表示“本翻译单元内链接，对外不可见”；在函数内表示局部静态存储期；在类内表示静态成员，全类共有一份，需在类外定义（除 const 整型等可类内初始化）。**namespace** 把名字包在命名空间中，避免全局命名冲突，可用 `namespace A { }`、`using`、`namespace alias` 等。此外 **extern** 用于声明在其他翻译单元定义的变量或避免重复定义；**thread_local** 表示线程局部存储；**constexpr** 表示编译期常量或可在编译期求值的函数，这些在需要时按文档和练习逐步掌握即可。

## 二、Memory Management（内存管理 — HFT 极重要）

### 1. 原始内存 API

**new/delete vs malloc/free**

**new** 在分配内存后调用构造函数，**delete** 在释放前调用析构函数，且是类型安全的（不用强转）、可重载的（类专属的 operator new/delete）、分配失败时抛 `std::bad_alloc`（除非用 nothrow 版本）。**malloc** 只分配指定字节数，不调用任何构造；**free** 只释放内存，不调用析构。因此用 malloc 得到的“裸内存”若放 C++ 对象，必须用 **placement new** 构造，并手动调用析构再 free。**绝不能混用**：new 出来的用 delete 释放，malloc 出来的用 free 释放；否则行为未定义。在 C++ 中应优先使用 new/delete 或（更好）智能指针与标准容器，减少手写 new/delete。

**Memory fragmentation**

频繁地小块分配和释放会在堆中留下许多“空洞”：总空闲量足够，但没有一块连续空间满足新的分配请求，导致分配失败或分配器被迫向系统申请更多页，甚至出现明明还有很多空闲却无法分配的情况。解决思路包括：减少对通用堆的依赖；对固定大小对象使用**内存池**或**对象池**，复用已释放的块；对短生命周期、成批创建的对象使用 **arena** 一次性分配、统一释放；以及复用对象而非反复 new/delete。在长时间运行、高吞吐或低延迟系统中，碎片会直接影响稳定性和延迟。

### 2. 智能指针

**Smart pointers 概览**

智能指针用 RAII 把堆上对象的生命周期绑在栈对象上：智能指针析构时自动释放所管理的对象（或减少引用计数）。标准库提供三种：**unique_ptr** 独占所有权，零开销；**shared_ptr** 共享所有权，引用计数；**weak_ptr** 不占所有权，用于打破循环或观察对象是否仍存在。使用智能指针能大幅减少忘记 delete、重复 delete、异常安全等问题，是现代 C++ 的首选方式。

**unique_ptr**

**unique_ptr** 独占所指向对象，不可拷贝，只能移动；析构时调用删除器（默认 `delete`）释放对象。大小通常等于一个指针，无额外计数等开销，适合“单一所有者”的堆对象、工厂函数返回值、Pimpl 等。可指定自定义删除器（如 `std::unique_ptr<FILE, decltype(&fclose)>`）以管理文件句柄等资源。C++14 起可用 `std::make_unique` 创建，避免手写 new。

**shared_ptr**

**shared_ptr** 通过引用计数共享所有权：每多一个 shared_ptr 指向同一对象，计数加一；某个 shared_ptr 析构时计数减一，减到零时删除对象。实现上通常还有控制块（计数、弱引用计数、删除器等）。注意：引用计数的增减是原子操作，有成本；应避免不必要的 shared_ptr 拷贝；若存在**循环引用**（A 持有 B 的 shared_ptr，B 又持有 A 的），计数永不为零，会泄漏，此时应把其中一侧改为 **weak_ptr**。

**weak_ptr**

**weak_ptr** 从 shared_ptr 构造，但**不增加**引用计数，不会阻止对象被析构。用 `lock()` 可得到 shared_ptr：若对象还活着则返回有效 shared_ptr，否则返回空。用于缓存（不阻止被缓存对象释放）、观察者、以及打破 shared_ptr 的循环引用。使用前必须检查 `lock()` 的返回值是否非空。

**Custom deleters**

智能指针的删除器在析构时调用，用于释放资源。unique_ptr 的删除器是类型的一部分（可无额外开销）；shared_ptr 的删除器存在控制块中（有类型擦除开销）。删除器可以是函数指针、lambda、`std::function` 等，适合管理 FILE*（用 fclose）、句柄、或把对象“还回”自定义池而不是 delete。这样所有资源都能用统一的“智能指针 + 删除器”方式管理。

**Cyclic reference issues**

若 A 持有 B 的 shared_ptr，B 又持有 A 的 shared_ptr，则两者引用计数至少为 1，永远不会变成 0，对象不会被释放，造成泄漏。解决办法：将其中一侧改为 **weak_ptr**（如 B 只持有 A 的 weak_ptr，需要时 lock 得到 shared_ptr），或重新设计所有权，让其中一方“拥有”另一方、另一方只观察或引用。在设计阶段就避免循环 shared_ptr 能省去很多调试。

### 3. 高级分配策略（与 HFT 相关）

**Placement new**

**Placement new** 的形式是 `new (address) T(args...)`，在已有的一块内存（address）上构造 T 类型的对象，不分配新内存。用于内存池（从池中取一块对齐的内存再构造）、对象池、共享内存、或需要精确控制对象地址的场景。用 placement new 构造的对象必须**手动**调用析构函数（如 `p->~T()`）再释放内存，因为 placement new 不负责分配，对应的“释放”也可能由池或 arena 统一处理。

**Memory pools**

**内存池**预先向系统申请一大块连续内存，然后按固定或可变大小从中切出块分配给请求者；释放时把块还回池中复用，而不是还给系统。这样减少对 malloc/free 的调用次数，降低碎片，分配延迟更稳定、可预测。适合分配大量同尺寸或少量几种尺寸的小对象；实现时需考虑对齐、线程安全（若多线程使用）、以及回收策略。

**Object pools**

**对象池**预先创建一批对象（或预留内存 + placement new），使用完后不析构，而是**重置状态**并放回池中供下次使用。这样避免频繁的构造/析构和堆分配，适合短生命周期、同类型、高吞吐的对象（如网络消息、订单、事件）。注意重置必须把对象恢复到合法状态，并考虑异常安全；若类型复杂，可能不如内存池 + placement new 灵活。

**Arena allocation**

**Arena**（有时叫 region 或 zone）是一块连续内存，分配时只在这块内存里顺序往后“切”，不单独释放某次分配，只在 arena 生命周期结束时整体释放（或重置指针重复使用）。分配操作只是指针前进，极快、无碎片、地址连续缓存友好。适合“同一阶段创建、一起释放”的数据（如解析一棵 AST、处理一帧数据）。在低延迟或实时系统中，arena 是常见的基础设施。

**Stack allocation optimizations**

在热路径上尽量让对象在**栈**上或使用**小对象优化**（如小字符串、小 vector 内嵌存储），减少堆分配和锁竞争（若堆分配器有全局锁）。可谨慎使用 `alloca` 在栈上要可变长空间（注意栈大小限制和可移植性）；或把固定大小的缓冲区、临时数组放在栈上；或为高频类型实现自定义的栈式分配器。栈上对象自动回收，无碎片，且与调用栈一起，缓存局部性通常更好。

**Cache-friendly allocation**

让频繁一起访问的数据在内存中**连续存放**、按**缓存行**对齐、避免**false sharing**（不同线程写同一缓存行的不同变量导致缓存行反复失效）。分配策略（池、arena、对象布局）会直接影响访问模式：顺序分配、顺序访问往往比随机分配、随机访问更友好，能提高 L1/L2/L3 命中率。在 HFT 或数值密集、延迟敏感代码中，缓存友好往往比“少几条指令”影响更大，需要结合 profiling 与数据布局设计。

**第一块（Core C++ Fundamentals）**：按 1 → 6 的顺序过一遍，重点把 RAII、值/引用/移动、多态与 vtable、Rule of 3/5/0 吃透，每个知识点配合小段代码验证（如写一个带资源的类、写一个多态层次、观察拷贝与移动的调用）。
